# ADR-0006: v0.1.x and v0.2.x retroactive decisions

* **Status:** Accepted — 2026-05-29 (retroactive consolidation)
* **Decision Makers:** Ganesh
* **Supersedes:** —
* **Last reviewed:** 2026-05-29

## Context and Problem Statement

Steno shipped v0.1.0 through v0.2.17 without ADRs. [ADR-0001](0001-record-architecture-decisions.md) introduced the practice. The biggest individual decisions in that earlier period each warrant their own ADR going forward, but writing six separate retroactive ADRs simultaneously is the kind of ceremony that ADR-0001 says not to do — small chance any of them get the depth they deserve.

This ADR is a consolidation: catch up on the six decisions in one document, in less detail than a freshly-written ADR would have, while preserving the rationale enough that future-self doesn't relitigate them. If any of these decisions get revisited later, they get promoted to their own ADR at that time.

## Decision A: STT engine — WhisperKit

**Considered:** WhisperKit (Argmax), Apple SpeechAnalyzer, Parakeet (NVIDIA), Qwen3-ASR, IndicWhisper, faster-whisper via Python bridge.

**Outcome:** WhisperKit. Selected for:
- Native Core ML — runs on Apple Neural Engine, no Python runtime.
- Multilingual including Indian languages (Hindi, Tamil) — Parakeet and Apple SpeechAnalyzer were English-first.
- Per-window language detection (`detectLanguage: true`) works for code-switching audio.
- Argmax maintains it actively; model formats are stable.

**Rejected alternatives:**
- **Apple SpeechAnalyzer:** macOS 26+ only at the time, English-only.
- **Parakeet:** Faster on benchmark but English-only and lower accuracy on Indian languages.
- **Python-bridged faster-whisper:** Adds runtime dependency, breaks the "no cloud, no daemon" promise.

**Where it lives in code:** `Steno/Services/TranscriptionEngine.swift` wraps WhisperKit. Model names use underscores (`large-v3_turbo`, not `large-v3-turbo`) — see AGENTS.md "WhisperKit Notes."

## Decision B: Speaker ID — FluidAudio

**Considered:** FluidAudio (FluidInference), pyannote.audio via Python, SpeakerNet (NVIDIA), 3D-Speaker (Alibaba).

**Outcome:** FluidAudio. Selected for:
- Native Core ML — same constraint as Decision A.
- LS-EEND model produces frame-level speaker labels usable for diarization.
- Reasonable latency for post-recording processing.

**Rejected alternatives:**
- **pyannote:** Python runtime dependency.
- **SpeakerNet, 3D-Speaker:** Not Core ML compatible without conversion work.

**Open issue:** Diarization quality on 3+ speaker calls is mediocre. VBx (Variational Bayes x-vector) was identified as a better algorithm but implementation deferred — tracked in backlog. Two-track recording (deferred from ADR-0004) would let FluidAudio operate on mic and system streams separately, sidestepping the multi-speaker-in-one-stream problem.

**Where it lives in code:** `Steno/Services/DiarizationManager.swift`.

## Decision C: License — BSL 1.1

**Considered:** MIT, Apache 2.0, GPL-3.0, AGPL-3.0, BSL 1.1, proprietary.

**Outcome:** BSL 1.1 (Business Source License) with Apache 2.0 change date.

**Rationale:**
- Permissive enough for personal/non-commercial use without friction.
- Restricts commercial competitors from forking and selling Steno as their own product during the "active commercial window."
- Auto-converts to Apache 2.0 after the change date (4 years) — preserves the open-source ethos long-term.
- MariaDB, Sentry, CockroachDB precedent — well-understood by developers and businesses.

**Rejected alternatives:**
- **MIT/Apache 2.0:** A larger competitor could fork, polish, and sell as their own with no obligation back. For a recording tool with a meaningful market, this is a real risk.
- **GPL/AGPL:** Copyleft scares away potential App Store distribution (and possibly even Homebrew tap acceptance depending on contributor policy).
- **Proprietary:** Loses the open-source community benefits (issue reports, contributors, the "no cloud, no telemetry" trust signal).

**Open question:** BSL 1.1 + Apple Developer Program agreement compatibility for App Store distribution. Verified informally as OK but worth re-verifying when steno-ios launches.

## Decision D: Clean-room engineering

**Rule (in AGENTS.md):** All code is written from Apple's documentation, framework APIs, and first principles. Do not copy implementations from other projects. Do not reference external apps, competitors, or third-party projects in code comments, commit messages, or documentation. When solving a problem, design from the platform APIs — not by mirroring how another app does it.

**Rationale:**
- Legal: a future commercial Steno can't be challenged on derivative-work grounds if there's no copied source.
- Pedagogical: forces understanding of *why* the API works the way it does, not just mimicry.
- Maintainability: code born from API understanding is easier to debug than code reverse-engineered from another app's behavior.

**Trade-off accepted:** Slower initial implementation. Some problems take research time that copying would skip. The v0.2.11–v0.2.14 incident's lesson — "research before implementing, not after shipping" — partly compensates by forcing the research up front.

## Decision E: Thread safety — NSLock over actors for audio threads

**Rule (in AGENTS.md "Thread Safety Rules"):** Audio IO threads (mic callback, system audio IO proc) run outside any actor. Code on these threads must:
1. Never access `self` of any `@MainActor` or actor-isolated class.
2. Use explicit local variable captures in closures (not capture lists with `self`).
3. Protect shared mutable state with NSLock (`AudioSharedState`, `AudioFileWriter`, `StreamingTranscriber`).
4. Keep lock-hold time minimal — do work outside the lock, lock only for reads/writes.

**Rationale:**
- Swift 6 actors require async access. Audio callbacks are sync, real-time, deadline-pressured. Actor-isolated state can't be reached from a Core Audio IO proc without async hops, and async hops violate the real-time deadline.
- NSLock works in sync contexts. It's the only viable cross-thread synchronization for audio threads.
- This rule was learned the hard way through actor-isolation crashes in v0.1.0 → v0.1.3 (commits `108c78c`, `8e5ecf1`, `978a036`, `4f5b089`).

**Trade-off accepted:** Less Swift-6-idiomatic. Modern Swift code prefers actors; audio code can't.

## Decision F: Action-side-effect checklist

**Rule (in AGENTS.md):** Before shipping any view change, trace every user action through the full loop:
1. **Every Button:** what happens on success? On failure? Does the user see both?
2. **Every Toggle:** does the change take effect immediately? Is the system it controls notified?
3. **Every async action:** is there a loading state? What if the view disappears mid-task?
4. **Every onChange:** should it fire on every change, or only on commit?
5. **Every delete/remove:** does it clean up all related state (files, index, selection)?

**The pattern: trigger → state change → side effect → user feedback → error path.**

**Rationale:**
- The four action-side-effect bugs found in the v0.2.1 full UI audit (commit `ab64258`) were all of one shape: action triggered but no user feedback or no error path.
- The checklist makes the failure mode explicit so it's caught before ship.

**Where it lives:** `AGENTS.md` "Action-Side-Effect Checklist" section.

## Consequences

**Good:**
- All six decisions are now searchable. A future session asking "why WhisperKit?" finds this file.
- The retroactive consolidation pattern is itself a precedent — for future bursts of small unrecorded decisions, write one consolidation ADR rather than six separate ones.

**Bad:**
- Each decision gets less depth than a freshly-written ADR would have. Some considered alternatives may be misremembered or compressed.
- If one of these decisions gets revisited deeply later, this file needs splitting — promote the relevant section to its own ADR and supersede it here.
